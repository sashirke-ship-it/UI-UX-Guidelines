## Design documentation model

1. **Catalog → categorize → specify:** bullet inventory of UI elements → categories → detailed guidelines per category (anatomy, spacing, states, do/don't).
2. **Reference frameworks:** NN/g + Material for sizing, density, pattern baselines; aosenuma tokens + ShadCN are **binding** for implementation.
3. **Visual-first handoff:** gradients, hover/active, spacing on **components** in canonical Figma (swatches alone insufficient).
4. **Context-specific hierarchy:** document density/type in Figma per pattern.

## Rollout plan (phased)

**Phase A — Inventory:** list all UI elements; group into categories; assign draft owner per category.

**Phase B — Visual color and gradients:** Figma on real components; semantic messaging illustrated; red reserved for negative semantics.

**Phase C — Component specs:** buttons first (all states + padding callouts) → extend to high-traffic components.

**Phase D — Written rules:** per-category do/don't + anatomy; each links to Figma frames.

**Phase E — Stakeholder follow-up:** open questions resolved; visual acceptance before "complete".

## Standard workflow (rinse-and-repeat)

1. **Intake** — requirements, screen goal, user type, schema, actions/states, visual acceptance inputs.
2. **Generate (AI pass 1)** — prompt recipes below; token-accurate Figma into Cursor before codegen.
3. **Validate** — tokens, states, accessibility baseline.
4. **Iterate (AI pass 2)** — targeted deltas only.
5. **Ship** — matches tokens/patterns; ShadCN; required states.

## AI prompt recipes (§ 16)

### Prompt A — New screen layout

- Use this repo `design.md` as hard constraints; ShadCN + Tailwind tokens.
- Inputs: screen goal, user type, data schema, required actions, required states.
- Output: component tree, interaction notes, token usage notes, acceptance checklist.

### Prompt B — Component generation/modification

- Inputs: component name, usage, variants, constraints.
- Output: props interface, variants, styling notes, example usage.
