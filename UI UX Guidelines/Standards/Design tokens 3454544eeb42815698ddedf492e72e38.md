# Design tokens

## Why tokens

Tokens create **one vocabulary** for design and code. Prefer **semantic** names over raw values.

## Naming convention (example)

`{category}.{role}.{variant}` — e.g. `color.text.primary`, `space.stack.md`, `radius.md`.

## Core token groups

- **color.** semantic text, surface, border, accent, status (success, warning, danger, info)
- **space.** stack, inline, inset, squish-inset
- **radius.** none, sm, md, lg, pill
- **shadow.** sm, md, lg (or none)
- **type.** family, size, weight, line-height
- **motion.** duration, easing

## Dark mode (if applicable)

- Pair **semantic** tokens to light/dark values.
- Never hard-code hex in components; always reference semantic tokens.

## Code mapping

Document how tokens map to implementation:

- **CSS variables:** `--color-text-primary`
- **Theme object:** `theme.colors.text.primary`
- **Platform:** Swift `Color.textPrimary` — align names across stacks.

## Contribution

- Token changes require **engineering + design** review and a **changelog** entry.
- Deprecate for one release cycle before removal when possible.
